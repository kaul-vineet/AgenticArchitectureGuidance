## 🧠 Key Architectural Principles

*Consolidated from DeepLearning.AI, HuggingFace / MAST, smolagents, Microsoft CAF, Azure Architecture Center, and Azure Well-Architected Framework:*

| # | Principle | Source |
|---|---|---|
| 1️⃣ | **Minimise LLM calls**<br>*Every generative step compounds error probability; route deterministic tasks to deterministic code.*<br>A 5-hop chain with 95% per-step reliability delivers only 77% end-to-end reliability — deterministic routing eliminates this compounding entirely. PA flows and CS Topics handle all structured tool calls without LLM involvement. Target: fewer than 20% of interactions involve generative processing. | [HuggingFace / smolagents](https://huggingface.co/blog/smolagents) |
| 2️⃣ | **Default to Flows, escalate to Crews**<br>*Use deterministic sequential flows for ~80% of tasks; reserve autonomous agents for genuinely open-ended work.*<br>Structured flows have 100% reliability versus 30% for multi-agent collaboration. The decision to elevate a task to a crew requires explicit justification — not convenience or team preference. Every Foundry build must first demonstrate that CS could not reliably deliver the outcome. | [DeepLearning.AI](https://www.deeplearning.ai/the-batch/how-agents-can-improve-llm-performance/) |
| 3️⃣ | **Clarification before action**<br>*Resolve ambiguous intent before routing — acting on unspecified input guarantees wrong outcomes.*<br>OpenEnv/Turing research shows a 50pp accuracy collapse (90% → 40%) when the same tasks are worded ambiguously versus explicitly. A single disambiguation question eliminates this entire failure class. Ambiguity causes confident wrong answers with no error signal to the user — the most operationally dangerous failure mode. | [MAST (FM-2.2)](https://huggingface.co/blog/ibm-research/itbenchandmast) |
| 4️⃣ | **External verification — never self-check**<br>*A separate tool must confirm every write; the model that generated an error cannot reliably validate it.*<br>LLMs exhibit systematic confirmation bias — they are significantly more likely to validate their own output than an external check would. The verifier must be a deterministic system read-back, not a second LLM call on the same context. MAST FM-3.3 identifies self-verification as a leading cause of undetected write failures. | [MAST (FM-3.3)](https://huggingface.co/blog/ibm-research/itbenchandmast) |
| 5️⃣ | **Externalize termination logic**<br>*Max steps, timeouts, and stop conditions live in code — never in the LLM's own reasoning.*<br>An LLM cannot reliably count to 10. Without hard-coded termination, runaway agents exhaust context windows, generate infinite loops, and accumulate uncapped costs. MAST FM-1.5 and FM-3.1 document missing or LLM-delegated termination as a top enterprise failure mode. Termination conditions must be declared before build begins. | [MAST (FM-1.5, FM-3.1)](https://huggingface.co/blog/ibm-research/itbenchandmast) |
| 6️⃣ | **Tool argument quality is the #1 failure cause**<br>*Typed schemas, example calls, and structured error responses prevent the majority of agentic failures.*<br>IBM AssetOpsBench found a 33pp gap between top-performing agents (94% tool accuracy) and low performers (61%) — larger than any model capability or prompt quality difference. Root causes were universally argument quality: missing example calls, untyped parameters, vague field names, absent structured error responses. Every connector must have all four before promotion. | [HuggingFace / OpenEnv](https://huggingface.co/blog/openenv-turing) |
| 7️⃣ | **Consolidate tools**<br>*Two calls that always execute in sequence with no branch between them should be one tool.*<br>Each unnecessary tool boundary adds a network call, a retry surface, and token cost. Sequential chaining without decision points is deterministic and must be implemented as a single atomic operation. Exception: steps that require independent retry logic or individual verification may remain separate. | [HuggingFace / smolagents](https://huggingface.co/blog/smolagents) |
| 8️⃣ | **Multi-agent is a reliability tax**<br>*Each agent boundary multiplies error probability; add one only where isolation is genuinely required.*<br>Single-agent to multi-agent handoff costs approximately 10–15 accuracy points (68% → 47% per AssetOpsBench). The only valid justifications are: different data sovereignty, separate compliance scope, or connectors that cannot be shared. Organisational convenience is never a valid reason for a new boundary. | [HuggingFace / AssetOpsBench](https://huggingface.co/blog/ibm-research/assetopsbench-playground-on-hugging-face) |
| 9️⃣ | **Context hygiene at scale**<br>*Pass only what the downstream agent needs — bloated context degrades accuracy and leaks state.*<br>MAST FM-1.4 shows 24% of traces in overloaded systems lose conversation history mid-session versus 0% for focused agents. State propagation must be typed and explicit at every boundary — never the full chat history. Context scope must be defined per boundary in the Agent Design Template before build begins. | [HuggingFace / Forge RL](https://huggingface.co/blog/dcarpintero/design-patterns-for-building-agentic-workflows) |
| 🔟 | **Sensitive state in side channel**<br>*Credentials, PII, and secrets never enter the conversation context; fetch at execution time only.*<br>Conversation context can be logged, traced, or inadvertently surfaced in follow-up responses. Secrets in context are a data exfiltration risk that no content safety filter can fully mitigate. All credentials must live in Power Platform connection references or Azure Key Vault — never in Topics, system prompts, or Workflow YAML. | [HuggingFace / Neuro SAN](https://huggingface.co/blog/danyoung/neuro-san-is-all-you-need) |
| 1️⃣1️⃣ | **Tool accuracy > 90% as deployment gate**<br>*Back-calculate required per-step accuracy from target end-to-end accuracy before build begins.*<br>A 3-hop workflow targeting 98% end-to-end accuracy requires each individual step to succeed at 99.3% or better. Teams that skip this calculation consistently promote agents that perform acceptably in isolation but fail in production chains. Required per-step accuracy must be computed in the Agent Design Template for each domain before build begins. | [HuggingFace / AssetOpsBench](https://huggingface.co/blog/ibm-research/assetopsbench-playground-on-hugging-face) |
| 1️⃣2️⃣ | **Human-in-the-loop for irreversible actions**<br>*Classify every action as Advisory, Supervised, or Autonomous; human review gates all irreversible writes.*<br>The three-tier classification provides a decision framework for every action type before build begins: Advisory (agent recommends, human decides), Supervised (agent executes, 15-min undo window), Autonomous (no review required). Irreversible writes without a review gate are the highest-risk category — the agent cannot undo what it does not know is wrong. | [DeepLearning.AI](https://www.deeplearning.ai/the-batch/how-agents-can-improve-llm-performance/) · [Microsoft CAF](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/governance-security-across-organization) |
| 1️⃣3️⃣ | **Use lowest complexity that reliably works**<br>*PA flow → CS Topic → specialist agent → Foundry; document failure at each rung before escalating.*<br>Complexity escalation must be evidence-based, not architectural preference. Each rung adds latency, cost, and failure surface. CS is the default platform; Foundry is the exception — every Foundry build requires documented evidence that CS could not meet the requirement. No rung may be skipped without documented justification. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) |
| 1️⃣4️⃣ | **Justify every agent boundary explicitly**<br>*Three valid reasons only: security isolation, prompt overflow, or tool overload. Preference is not a reason.*<br>Agent boundaries are structural commitments — they add maintenance surface, complicate testing, and introduce handoff costs that compound at scale. Valid reasons: (a) security or compliance isolation required, (b) prompt complexity exceeds 8,000 characters, (c) tool count exceeds ~15. The default is no new boundary; the burden of proof rests with the team proposing one. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) |
| 1️⃣5️⃣ | **Single responsibility per agent**<br>*Each agent has a written job description and an explicit "does NOT handle" list.*<br>Agents without a "does NOT handle" list accumulate scope through organic requests — the software equivalent of scope creep. Monthly intent overlap reviews catch drift before it becomes structural. Overlapping agents are consolidated, not left running in parallel; redundant invocations detected by App Insights trigger a consolidation review. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [Microsoft CAF](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/governance-security-across-organization) |
| 1️⃣6️⃣ | **Right-size the model per agent role**<br>*Gateway: classification model. Domain: reasoning model. Specialist: smallest that passes the capability test.*<br>Model selection is a cost and reliability decision, not a quality-maximization one. Running the largest available model for routing and classification is waste with no quality benefit. Running too small a model for complex reasoning is a quality risk. Model selection and capability test result must be recorded in the Agent Design Template for each specialist agent. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) |
| 1️⃣7️⃣ | **Validate output before passing downstream**<br>*Schema-validate every inter-agent handoff in code; malformed output halts the workflow.*<br>A downstream agent that receives malformed input will propagate the error silently or produce a confident wrong answer. Validation in code at each boundary makes failures loud, early, and traceable. Validation must be enforced in code on both platforms — never delegated to LLM judgment. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) |
| 1️⃣8️⃣ | **Surface errors explicitly — never swallow failures**<br>*Every failure returns a typed error with code, field, and next steps — never a silent null or vague message.*<br>Silent failures are operationally dangerous — they produce a response that looks successful while the underlying action did not complete. Typed errors with structured fields (`errorCode`, `field`, `expectedFormat`, `retryable`) enable automated retry, human escalation, and a complete audit trail. Users see either a confirmed success with a reference number or a named error with next steps — never a silent failure. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) |
| 1️⃣9️⃣ | **Idempotency for all writes**<br>*Every write carries a correlationId-based key; retry cannot produce duplicate records.*<br>Network failures and agent retries are expected in distributed systems. Without idempotency keys, a leave submission retried after a timeout creates two confirmed requests — one invisible to the user. The idempotency key pattern `{correlationId}-{actionType}-{date}` is a required field in the Agent Design Template for every write action, on both CS and Foundry platforms. | [Microsoft Tech Community](https://techcommunity.microsoft.com/blog/educatordeveloperblog/ai-didn%E2%80%99t-break-your-production-%E2%80%94-your-architecture-did/4482848) |
| 2️⃣0️⃣ | **Kill switch — disable tools without redeployment**<br>*Agent, tool, and fleet — each disableable independently without code changes or redeployment.*<br>Production incidents require minutes to mitigate, not a CI/CD pipeline run. Three levels of granularity are required: agent-level (immediate portal disable), tool-level (specific connector disabled; agent degrades gracefully to read-only), fleet-level (maintenance broadcast to all agents simultaneously). The kill switch runbook for each domain must be written before go-live — not drafted after an incident. | [Microsoft CAF](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/governance-security-across-organization) |
| 2️⃣1️⃣ | **Security trimming in every agent, not just the edge**<br>*RLS and RBAC enforce data entitlement at the data layer; the LLM never decides what a user can see.*<br>Delegating access control to the LLM ("only show data the user is allowed to see") is not access control — it is a prompt instruction that can be bypassed via injection or adversarial inputs. Security trimming must be enforced at the data layer (Dataverse RLS, Azure Search security filters), not the reasoning layer. No agent response may contain data not returned by the underlying system for that user's entitlement. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) |
| 2️⃣2️⃣ | **Content safety at every layer**<br>*Safety filters apply at user input, tool response ingestion, and agent output — not only at the edge.*<br>A single edge filter can be bypassed by injecting malicious content through a tool response, a retrieval result, or an API call. Layered safety at input, tool response ingestion, and output makes end-to-end injection attacks significantly harder to construct. Safety filter level is configurable per domain — stricter for HR and Finance — and must be set before go-live. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) |
| 2️⃣3️⃣ | **Scope agent authority to operations, not tools**<br>*PA connectors are scoped to specific operations; blast radius is capped at the infrastructure layer.*<br>Granting broad API access and relying on the agent's reasoning to stay within scope is not a security posture — it is a hope. Infrastructure-layer scoping caps the blast radius of any misconfiguration, injection, or model error. Write operations are capped at 10 per session per specialist agent; scopes are enforced at the infrastructure layer, never declared in the system prompt. | [Microsoft Tech Community](https://techcommunity.microsoft.com/blog/educatordeveloperblog/ai-didn%E2%80%99t-break-your-production-%E2%80%94-your-architecture-did/4482848) |
| 2️⃣4️⃣ | **Instrument all handoffs, not just final output**<br>*CorrelationId propagates to every agent call, tool invocation, and PA flow — one ID reconstructs any trace.*<br>Final output logging tells you what the agent returned but not why it went wrong or where. Without per-hop tracing, a 5-hop failure becomes a debugging exercise across five disconnected logs with no common key. A single correlationId must reconstruct the complete end-to-end trace across all agents and platforms — this is a non-negotiable acceptance criterion for every domain. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [Microsoft Azure Blog](https://azure.microsoft.com/en-us/blog/agent-factory-top-5-agent-observability-best-practices-for-reliable-ai/) |
| 2️⃣5️⃣ | **Automated evaluation in CI/CD**<br>*Every solution import or prompt change triggers a 50-sample eval harness before promotion.*<br>A prompt is code. Text changes to an agent's system prompt can degrade routing accuracy, introduce hallucinations, or break downstream schema expectations — with no visible diff in the repository unless eval gates enforce it. LLM-as-Judge is the scoring method (exact-match is not used). A prompt change has the same promotion gate as a code change, on both CS and Foundry platforms. | [Microsoft Azure Blog](https://azure.microsoft.com/en-us/blog/agent-factory-top-5-agent-observability-best-practices-for-reliable-ai/) |
| 2️⃣6️⃣ | **Evaluate continuously in production**<br>*Weekly KPIs plus monthly full eval re-run catch silent drift from model updates and input distribution shift.*<br>Model providers update base models without notice. Input distribution shifts as users change behavior over time. Neither change produces an error — they produce a silent quality degradation that only continuous evaluation can surface. A greater than 5% week-over-week drop in intent resolution rate, connector success rate, or session completion triggers an immediate review ticket. | [Microsoft Azure Blog](https://azure.microsoft.com/en-us/blog/agent-factory-top-5-agent-observability-best-practices-for-reliable-ai/) |
| 2️⃣7️⃣ | **Maintain agent inventory — shadow agents are a risk**<br>*Every agent registered in Agent 365 before deployment; unregistered agents are treated as security incidents.*<br>Shadow agents are ungoverned agents: no named owner, no security review, no decommission plan, and no audit trail. At enterprise scale, ungoverned agents become the highest-probability path for data exfiltration and capability abuse. Agent 365 is the system of record; monthly CoE reviews flag agents unused for 30 days or unreviewed for 90 days. | [Microsoft CAF](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/governance-security-across-organization) |
| 2️⃣8️⃣ | **Externalize and version-control prompts**<br>*System prompts live in environment variables or App Configuration — a prompt change is a code deployment.*<br>Hardcoded prompts in Topics or Workflow YAML create invisible production changes that bypass review, testing, and rollback. Environment variables and App Configuration provide diff history, reviewer approval, and instant rollback — the same guarantees as a code change. Every prompt change requires a PR with reviewer sign-off and must pass the eval harness before merge. | [Azure Well-Architected Framework](https://learn.microsoft.com/en-us/azure/well-architected/ai/application-design) |
