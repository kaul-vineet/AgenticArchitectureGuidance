## 🧠 Key Architectural Principles

*Consolidated from DeepLearning.AI, HuggingFace / MAST / IBM Research, smolagents, Microsoft CAF, Azure Architecture Center, Azure Well-Architected Framework, Google / DeepMind, and research by Andrej Karpathy, Ilya Sutskever, Sebastian Raschka, and Noam Shazeer. Principles 1–28 are the original foundational set. Principles 29–62 incorporate all Design Considerations (DC-1 through DC-31) and additional guidance validated from the Architecture References corpus — filtered for SaaS/PaaS applicability.*

| # | Principle | Source |
|---|---|---|
| 1️⃣ | **Minimise LLM calls**<br>*Every generative step compounds error probability; route deterministic tasks to deterministic code.*<br>A 5-hop chain with 95% per-step reliability delivers only 77% end-to-end reliability — deterministic routing eliminates this compounding entirely. PA flows and CS Topics handle all structured tool calls without LLM involvement. Target: fewer than 20% of interactions involve generative processing. | [HuggingFace / smolagents](https://huggingface.co/blog/smolagents)  · [→ P1](./section-10#p1) |
| 2️⃣ | **Default to Flows, escalate to Crews**<br>*Use deterministic sequential flows for ~80% of tasks; reserve autonomous agents for genuinely open-ended work.*<br>Structured flows have 100% reliability versus 30% for multi-agent collaboration. The decision to elevate a task to a crew requires explicit justification — not convenience or team preference. Every Foundry build must first demonstrate that CS could not reliably deliver the outcome. | [DeepLearning.AI](https://www.deeplearning.ai/the-batch/how-agents-can-improve-llm-performance/)  · [→ P2](./section-10#p2) |
| 3️⃣ | **Clarification before action**<br>*Resolve ambiguous intent before routing — acting on unspecified input guarantees wrong outcomes.*<br>OpenEnv/Turing research shows a 50pp accuracy collapse (90% → 40%) when the same tasks are worded ambiguously versus explicitly. A single disambiguation question eliminates this entire failure class. Ambiguity causes confident wrong answers with no error signal to the user — the most operationally dangerous failure mode. | [MAST (FM-2.2)](https://huggingface.co/blog/ibm-research/itbenchandmast)  · [→ P3](./section-10#p3) |
| 4️⃣ | **External verification — never self-check**<br>*A separate tool must confirm every write; the model that generated an error cannot reliably validate it.*<br>LLMs exhibit systematic confirmation bias — they are significantly more likely to validate their own output than an external check would. The verifier must be a deterministic system read-back, not a second LLM call on the same context. MAST FM-3.3 identifies self-verification as a leading cause of undetected write failures. | [MAST (FM-3.3)](https://huggingface.co/blog/ibm-research/itbenchandmast)  · [→ P4](./section-10#p4) |
| 5️⃣ | **Externalize termination logic**<br>*Max steps, timeouts, and stop conditions live in code — never in the LLM's own reasoning.*<br>An LLM cannot reliably count to 10. Without hard-coded termination, runaway agents exhaust context windows, generate infinite loops, and accumulate uncapped costs. MAST FM-1.5 and FM-3.1 document missing or LLM-delegated termination as a top enterprise failure mode. Termination conditions must be declared before build begins. | [MAST (FM-1.5, FM-3.1)](https://huggingface.co/blog/ibm-research/itbenchandmast)  · [→ P5](./section-10#p5) |
| 6️⃣ | **Tool argument quality is the #1 failure cause**<br>*Typed schemas, example calls, and structured error responses prevent the majority of agentic failures.*<br>IBM AssetOpsBench found a 33pp gap between top-performing agents (94% tool accuracy) and low performers (61%) — larger than any model capability or prompt quality difference. Root causes were universally argument quality: missing example calls, untyped parameters, vague field names, absent structured error responses. Every connector must have all four before promotion. | [HuggingFace / OpenEnv](https://huggingface.co/blog/openenv-turing) · [→ P6](./section-10#p6) |
| 7️⃣ | **Consolidate tools**<br>*Two calls that always execute in sequence with no branch between them should be one tool.*<br>Each unnecessary tool boundary adds a network call, a retry surface, and token cost. Sequential chaining without decision points is deterministic and must be implemented as a single atomic operation. Exception: steps that require independent retry logic or individual verification may remain separate. | [HuggingFace / smolagents](https://huggingface.co/blog/smolagents) · [→ P7](./section-10#p7) |
| 8️⃣ | **Multi-agent is a reliability tax**<br>*Each agent boundary multiplies error probability; add one only where isolation is genuinely required.*<br>Single-agent to multi-agent handoff costs approximately 10–15 accuracy points (68% → 47% per AssetOpsBench). The only valid justifications are: different data sovereignty, separate compliance scope, or connectors that cannot be shared. Organisational convenience is never a valid reason for a new boundary. | [HuggingFace / AssetOpsBench](https://huggingface.co/blog/ibm-research/assetopsbench-playground-on-hugging-face) · [→ P8](./section-10#p8) |
| 9️⃣ | **Context hygiene at scale**<br>*Pass only what the downstream agent needs — bloated context degrades accuracy and leaks state.*<br>MAST FM-1.4 shows 24% of traces in overloaded systems lose conversation history mid-session versus 0% for focused agents. State propagation must be typed and explicit at every boundary — never the full chat history. Context scope must be defined per boundary in the Agent Design Template before build begins. | [HuggingFace / Forge RL](https://huggingface.co/blog/dcarpintero/design-patterns-for-building-agentic-workflows) · [→ P9](./section-10#p9) |
| 🔟 | **Sensitive state in side channel**<br>*Credentials, PII, and secrets never enter the conversation context; fetch at execution time only.*<br>Conversation context can be logged, traced, or inadvertently surfaced in follow-up responses. Secrets in context are a data exfiltration risk that no content safety filter can fully mitigate. All credentials must live in Power Platform connection references or Azure Key Vault — never in Topics, system prompts, or Workflow YAML. | [HuggingFace / Neuro SAN](https://huggingface.co/blog/danyoung/neuro-san-is-all-you-need) · [→ P10](./section-10#p10) |
| 1️⃣1️⃣ | **Tool accuracy > 90% as deployment gate**<br>*Back-calculate required per-step accuracy from target end-to-end accuracy before build begins.*<br>A 3-hop workflow targeting 98% end-to-end accuracy requires each individual step to succeed at 99.3% or better. Teams that skip this calculation consistently promote agents that perform acceptably in isolation but fail in production chains. Required per-step accuracy must be computed in the Agent Design Template for each domain before build begins. | [HuggingFace / AssetOpsBench](https://huggingface.co/blog/ibm-research/assetopsbench-playground-on-hugging-face) · [→ P11](./section-10#p11) |
| 1️⃣2️⃣ | **Human-in-the-loop for irreversible actions**<br>*Classify every action as Advisory, Supervised, or Autonomous; human review gates all irreversible writes.*<br>The three-tier classification provides a decision framework for every action type before build begins: Advisory (agent recommends, human decides), Supervised (agent executes, 15-min undo window), Autonomous (no review required). Irreversible writes without a review gate are the highest-risk category — the agent cannot undo what it does not know is wrong. | [DeepLearning.AI](https://www.deeplearning.ai/the-batch/how-agents-can-improve-llm-performance/) · [Microsoft CAF](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/governance-security-across-organization) · [→ P12](./section-10#p12) |
| 1️⃣3️⃣ | **Use lowest complexity that reliably works**<br>*PA flow → CS Topic → specialist agent → Foundry; document failure at each rung before escalating.*<br>Complexity escalation must be evidence-based, not architectural preference. Each rung adds latency, cost, and failure surface. CS is the default platform; Foundry is the exception — every Foundry build requires documented evidence that CS could not meet the requirement. No rung may be skipped without documented justification. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [→ P13](./section-10#p13) |
| 1️⃣4️⃣ | **Justify every agent boundary explicitly**<br>*Three valid reasons only: security isolation, prompt overflow, or tool overload. Preference is not a reason.*<br>Agent boundaries are structural commitments — they add maintenance surface, complicate testing, and introduce handoff costs that compound at scale. Valid reasons: (a) security or compliance isolation required, (b) prompt complexity exceeds 8,000 characters, (c) tool count exceeds ~15. The default is no new boundary; the burden of proof rests with the team proposing one. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [→ P14](./section-10#p14) |
| 1️⃣5️⃣ | **Single responsibility per agent**<br>*Each agent has a written job description and an explicit "does NOT handle" list.*<br>Agents without a "does NOT handle" list accumulate scope through organic requests — the software equivalent of scope creep. Monthly intent overlap reviews catch drift before it becomes structural. Overlapping agents are consolidated, not left running in parallel; redundant invocations detected by App Insights trigger a consolidation review. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [Microsoft CAF](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/governance-security-across-organization) · [→ P15](./section-10#p15) |
| 1️⃣6️⃣ | **Right-size the model per agent role**<br>*Gateway: classification model. Domain: reasoning model. Specialist: smallest that passes the capability test.*<br>Model selection is a cost and reliability decision, not a quality-maximization one. Running the largest available model for routing and classification is waste with no quality benefit. Running too small a model for complex reasoning is a quality risk. Model selection and capability test result must be recorded in the Agent Design Template for each specialist agent. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [→ P16](./section-10#p16) |
| 1️⃣7️⃣ | **Validate output before passing downstream**<br>*Schema-validate every inter-agent handoff in code; malformed output halts the workflow.*<br>A downstream agent that receives malformed input will propagate the error silently or produce a confident wrong answer. Validation in code at each boundary makes failures loud, early, and traceable. Validation must be enforced in code on both platforms — never delegated to LLM judgment. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [→ P17](./section-10#p17) |
| 1️⃣8️⃣ | **Surface errors explicitly — never swallow failures**<br>*Every failure returns a typed error with code, field, and next steps — never a silent null or vague message.*<br>Silent failures are operationally dangerous — they produce a response that looks successful while the underlying action did not complete. Typed errors with structured fields (`errorCode`, `field`, `expectedFormat`, `retryable`) enable automated retry, human escalation, and a complete audit trail. Users see either a confirmed success with a reference number or a named error with next steps — never a silent failure. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [→ P18](./section-10#p18) |
| 1️⃣9️⃣ | **Idempotency for all writes**<br>*Every write carries a correlationId-based key; retry cannot produce duplicate records.*<br>Network failures and agent retries are expected in distributed systems. Without idempotency keys, a leave submission retried after a timeout creates two confirmed requests — one invisible to the user. The idempotency key pattern `{correlationId}-{actionType}-{date}` is a required field in the Agent Design Template for every write action, on both CS and Foundry platforms. | [Microsoft Tech Community](https://techcommunity.microsoft.com/blog/educatordeveloperblog/ai-didn%E2%80%99t-break-your-production-%E2%80%94-your-architecture-did/4482848) · [→ P19](./section-10#p19) |
| 2️⃣0️⃣ | **Kill switch — disable tools without redeployment**<br>*Agent, tool, and fleet — each disableable independently without code changes or redeployment.*<br>Production incidents require minutes to mitigate, not a CI/CD pipeline run. Three levels of granularity are required: agent-level (immediate portal disable), tool-level (specific connector disabled; agent degrades gracefully to read-only), fleet-level (maintenance broadcast to all agents simultaneously). The kill switch runbook for each domain must be written before go-live — not drafted after an incident. | [Microsoft CAF](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/governance-security-across-organization) · [→ P20](./section-10#p20) |
| 2️⃣1️⃣ | **Security trimming in every agent, not just the edge**<br>*RLS and RBAC enforce data entitlement at the data layer; the LLM never decides what a user can see.*<br>Delegating access control to the LLM ("only show data the user is allowed to see") is not access control — it is a prompt instruction that can be bypassed via injection or adversarial inputs. Security trimming must be enforced at the data layer (Dataverse RLS, Azure Search security filters), not the reasoning layer. No agent response may contain data not returned by the underlying system for that user's entitlement. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [→ P21](./section-10#p21) |
| 2️⃣2️⃣ | **Content safety at every layer**<br>*Safety filters apply at user input, tool response ingestion, and agent output — not only at the edge.*<br>A single edge filter can be bypassed by injecting malicious content through a tool response, a retrieval result, or an API call. Layered safety at input, tool response ingestion, and output makes end-to-end injection attacks significantly harder to construct. Safety filter level is configurable per domain — stricter for HR and Finance — and must be set before go-live. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [→ P22](./section-10#p22) |
| 2️⃣3️⃣ | **Scope agent authority to operations, not tools**<br>*PA connectors are scoped to specific operations; blast radius is capped at the infrastructure layer.*<br>Granting broad API access and relying on the agent's reasoning to stay within scope is not a security posture — it is a hope. Infrastructure-layer scoping caps the blast radius of any misconfiguration, injection, or model error. Write operations are capped at 10 per session per specialist agent; scopes are enforced at the infrastructure layer, never declared in the system prompt. | [Microsoft Tech Community](https://techcommunity.microsoft.com/blog/educatordeveloperblog/ai-didn%E2%80%99t-break-your-production-%E2%80%94-your-architecture-did/4482848) · [→ P23](./section-10#p23) |
| 2️⃣4️⃣ | **Instrument all handoffs, not just final output**<br>*CorrelationId propagates to every agent call, tool invocation, and PA flow — one ID reconstructs any trace.*<br>Final output logging tells you what the agent returned but not why it went wrong or where. Without per-hop tracing, a 5-hop failure becomes a debugging exercise across five disconnected logs with no common key. A single correlationId must reconstruct the complete end-to-end trace across all agents and platforms — this is a non-negotiable acceptance criterion for every domain. | [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [Microsoft Azure Blog](https://azure.microsoft.com/en-us/blog/agent-factory-top-5-agent-observability-best-practices-for-reliable-ai/) · [→ P24](./section-10#p24) |
| 2️⃣5️⃣ | **Automated evaluation in CI/CD**<br>*Every solution import or prompt change triggers a 50-sample eval harness before promotion.*<br>A prompt is code. Text changes to an agent's system prompt can degrade routing accuracy, introduce hallucinations, or break downstream schema expectations — with no visible diff in the repository unless eval gates enforce it. LLM-as-Judge is the scoring method (exact-match is not used). A prompt change has the same promotion gate as a code change, on both CS and Foundry platforms. | [Microsoft Azure Blog](https://azure.microsoft.com/en-us/blog/agent-factory-top-5-agent-observability-best-practices-for-reliable-ai/) · [→ P25](./section-10#p25) |
| 2️⃣6️⃣ | **Evaluate continuously in production**<br>*Weekly KPIs plus monthly full eval re-run catch silent drift from model updates and input distribution shift.*<br>Model providers update base models without notice. Input distribution shifts as users change behavior over time. Neither change produces an error — they produce a silent quality degradation that only continuous evaluation can surface. A greater than 5% week-over-week drop in intent resolution rate, connector success rate, or session completion triggers an immediate review ticket. | [Microsoft Azure Blog](https://azure.microsoft.com/en-us/blog/agent-factory-top-5-agent-observability-best-practices-for-reliable-ai/) · [→ P26](./section-10#p26) |
| 2️⃣7️⃣ | **Maintain agent inventory — shadow agents are a risk**<br>*Every agent registered in Agent 365 before deployment; unregistered agents are treated as security incidents.*<br>Shadow agents are ungoverned agents: no named owner, no security review, no decommission plan, and no audit trail. At enterprise scale, ungoverned agents become the highest-probability path for data exfiltration and capability abuse. Agent 365 is the system of record; monthly CoE reviews flag agents unused for 30 days or unreviewed for 90 days. | [Microsoft CAF](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/governance-security-across-organization) · [→ P27](./section-10#p27) |
| 2️⃣8️⃣ | **Externalize and version-control prompts**<br>*System prompts live in environment variables or App Configuration — a prompt change is a code deployment.*<br>Hardcoded prompts in Topics or Workflow YAML create invisible production changes that bypass review, testing, and rollback. Environment variables and App Configuration provide diff history, reviewer approval, and instant rollback — the same guarantees as a code change. Every prompt change requires a PR with reviewer sign-off and must pass the eval harness before merge. | [Azure Well-Architected Framework](https://learn.microsoft.com/en-us/azure/well-architected/ai/application-design) · [→ P28](./section-10#p28) |
| 2️⃣9️⃣ | **Gateway routing must be measured and bounded at scale**<br>*Route by intent taxonomy using a bounded decision tree — not open-ended LLM prompt matching against an unbounded agent list.*<br>At 8–12 domain agents the gateway prompt approaches the CS 8,000-character instruction ceiling. Routing descriptions must be externalised to a Dataverse lookup and injected only for the matched intent at runtime. Gateway prompt token consumption must be measured at 6, 9, and 12 domain agents during the pilot wave before committing to a routing pattern — never assume the prompt will fit. | [Microsoft Copilot Studio — Agent Instructions](https://learn.microsoft.com/en-us/microsoft-copilot-studio/authoring-best-practices) · [MAST](https://huggingface.co/blog/ibm-research/itbenchandmast) · [→ P1](./section-10#p1) |
| 3️⃣0️⃣ | **Classify workflows as independent or dependent before routing**<br>*Cross-domain workflows with ordering constraints, shared state, or compensation requirements cannot be handled by conversational session orchestration.*<br>A CS session handles independent multi-domain calls correctly; dependent cross-domain workflows (where step A must complete before step B is permitted, or where step B failure requires step A rollback) require durable, stateful orchestration with compensation branches. Power Automate is the default orchestrator for dependent processes; Foundry Workflow for compliance-grade processes. A Workflow Registry in Dataverse — parallel to the Agent Registry — must document each named workflow with ordered steps, compensating actions, timeout policies, and correlationId prefix. | [Microsoft Copilot Studio — Quotas](https://learn.microsoft.com/en-us/microsoft-copilot-studio/requirements-quotas) · [Microsoft MARA](https://microsoft.github.io/multi-agent-reference-architecture/docs/reference-architecture/Reference-Architecture.html) · [→ P2](./section-10#p2) |
| 3️⃣1️⃣ | **Tool response sanitisation is mandatory for every user-writable system**<br>*Sanitisation between tool response and LLM context re-entry is a distinct and required layer — not covered by output content safety.*<br>Content safety at the output layer does not intercept malicious content embedded in tool responses from SharePoint documents, HRIS records, CRM notes, or Jira tickets. At 100 agents making tool calls across the enterprise the indirect injection attack surface is enterprise-wide. Every tool response from a user-writable system must pass through a sanitisation layer before entering the LLM context. This is a production blocker, not a recommendation — no domain goes live without a documented response sanitisation layer for all tool responses from user-writable systems. | [OWASP LLM Top 10 — LLM02](https://owasp.org/www-project-top-10-for-large-language-model-applications/) · [Azure Content Safety — Prompt Shield](https://learn.microsoft.com/en-us/azure/ai-services/content-safety/concepts/jailbreak-detection) · [→ P3](./section-10#p3) · [→ P15](./section-10#p15) |
| 3️⃣2️⃣ | **Preview features must not be the sole production dependency**<br>*Any component built on a preview feature must have a durable GA fallback designed in from the start.*<br>A 100-agent deployment over 6–12 months will ship before preview features reach GA. API or behaviour changes in preview can require rebuilding a component across the entire agent fleet. Wherever a preview feature is used, a durable alternative must be designed in parallel and kept deployable — the preview feature is the preferred path, not the only path. Preview-only dependencies must be flagged in the Agent Design Template with their documented fallback. | [Azure AI Foundry — Memory Feature (Preview)](https://learn.microsoft.com/en-us/azure/ai-services/agents/how-to/tools/azure-ai-memory) · [→ P4](./section-10#p4) |
| 3️⃣3️⃣ | **Platform capability constraints documented against principles — never silently dropped**<br>*Where a platform cannot implement a principle, record the constraint, the gap, and the mitigation explicitly in the Agent Design Template.*<br>Architectural Principle 16 (right-size model per role) cannot be implemented in Copilot Studio because CS locks to Microsoft-managed models. This is a named platform constraint, not a principle abandonment. For every principle a chosen platform cannot satisfy, the Agent Design Template must record the constraint, the reliability gap it creates, and the mitigation strategy — for example, routing capability-sensitive agents to Foundry instead of CS. Silent non-compliance with a principle is not an option. | [Azure Architecture Center — AI Agent Design Patterns](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [Microsoft Copilot Studio — Generative AI Features](https://learn.microsoft.com/en-us/microsoft-copilot-studio/nlu-gpt-overview) · [→ P5](./section-10#p5) |
| 3️⃣4️⃣ | **LLM-as-Judge calibrated to >80% human annotation agreement before use as an automated gate**<br>*Positional bias, verbosity bias, and self-enhancement bias produce false confidence in uncalibrated judges — recalibrate when the judge model is updated.*<br>Without calibration, the judge systematically over-rates certain output types and under-rates others. Calibration protocol: N≥200 samples per domain, annotated independently by at least 3 domain reviewers, targeting >80% Cohen's kappa agreement before the judge gates any production promotion. The judge model version must be tracked in the Agent Design Template; a judge model update requires recalibration before the updated judge is used. For domains where calibration cannot reach >80% agreement, fall back to ensemble judging (multiple model judges + 10% human spot-check). | [Sebastian Raschka — 4 Approaches to LLM Evaluation](https://magazine.sebastianraschka.com/p/llm-evaluation-4-approaches) · [MAST](https://huggingface.co/blog/ibm-research/itbenchandmast) · [→ P6](./section-10#p6) · [→ P23](./section-10#p23) |
| 3️⃣5️⃣ | **Circuit breaker per tool — defined degradation fallback before go-live**<br>*A tool that drops below the accuracy threshold in production must trigger automated fallback — continuous calling of a failing tool accumulates errors and corrupts downstream state.*<br>The 90% tool accuracy gate (Principle 11) is a deployment gate, not a runtime guarantee. Each tool must have a defined degradation mode in its Agent Design Template entry: what the agent does when the tool error rate exceeds a configurable threshold (default: fall back to read-only or FAQ-only mode). Error rate per tool per agent must be monitored in App Insights with an alert threshold and an automatic disablement trigger aligned with Principle 20 (kill switch). | [Azure Architecture Center — AI Agent Design Patterns](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [Microsoft CAF — Governance and Security](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/governance-security-across-organization) · [→ P7](./section-10#p7) |
| 3️⃣6️⃣ | **Typed error envelope propagated upstream without reinterpretation — only the gateway translates**<br>*Intermediate agents forward the typed error envelope untouched; translation to user-facing language happens once, at the gateway.*<br>In a 4-hop chain, intermediate agents that reinterpret errors introduce hallucination risk — the error message gains confidence and loses accuracy at each hop. The error envelope `{errorCode, errorMessage, retryable, agentId, stepId, correlationId}` must be defined in the Agent Design Template I/O contract for every agent. Intermediate agents forward; they never reinterpret. The gateway is the single translation point. This is a mandatory output field in every Agent Design Template, not a recommendation. | [MAST — Failure Taxonomy](https://huggingface.co/blog/ibm-research/itbenchandmast) · [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [→ P8](./section-10#p8) |
| 3️⃣7️⃣ | **correlationId is an enforced contract — responses missing it are rejected, not forwarded**<br>*In heterogeneous systems, correlationId propagation breaks silently without enforcement; it must be a required field in the A2A payload schema, not an optional header.*<br>The correlationId naming convention is only a reliability control when it is enforced. Gateway-level validation must reject any agent response missing the expected correlationId — it is treated as an error, not passed to the user or downstream. correlationId must be defined as a required field in the A2A payload schema. The correlationId propagation test is a go-live gate in the Agent Design Template Definition of Done — not a checkbox. | [A2A Protocol](https://a2a-protocol.org/latest/) · [Microsoft MARA](https://microsoft.github.io/multi-agent-reference-architecture/docs/reference-architecture/Reference-Architecture.html) · [→ P9](./section-10#p9) |
| 3️⃣8️⃣ | **Task-specific capability test required before model approval — benchmark scores are not sufficient**<br>*N=20 examples of the actual task type, scored against domain acceptance criteria, must pass before a model is approved for a role.*<br>LLMs have jagged capability profiles — a model that scores well on general benchmarks may fail on the specific subtask required. Model tier declaration in the Agent Design Template is not the same as validating that the selected model can reliably perform the specific operation. The capability test result (task type, N, pass rate, failure patterns) must be recorded in the Agent Design Template model selection section before build begins. "Seems capable" is not an acceptable model selection justification. | [Andrej Karpathy — Intro to Large Language Models](https://www.youtube.com/watch?v=zjkBMFhNj_g) · [State of GPT (Build 2023)](https://www.youtube.com/watch?v=bZQun8Y4L2A) · [→ P10](./section-10#p10) |
| 3️⃣9️⃣ | **Eval results report distribution statistics — pass rate, variance, and failure mode distribution — not just median**<br>*A high-variance result is not production-ready even at high median pass rate; distribution stability is the gate, not the median.*<br>A single successful eval run, or a single pass of the 50-sample harness, is one sample from the output distribution — not evidence of reliability. Results must report pass rate variance across 50 samples and across multiple prompt variants and model versions where applicable. Any domain where pass rate variance exceeds 10% across 50 samples is not ready for production regardless of median outcome. | [Andrej Karpathy — Intro to Large Language Models](https://www.youtube.com/watch?v=zjkBMFhNj_g) · [→ P11](./section-10#p11) |
| 4️⃣0️⃣ | **HITL review interface engineered as UX — a rubber-stamp approval flow is not a safety control**<br>*Each Adaptive Card approval must display action statement, reasoning summary, source data reference, and undo path — all visible without scrolling.*<br>A HITL step with inadequate UX — too much information, no visible undo path, time pressure — creates liability without safety. The approval-without-review rate (target: <5%) and actual review time (target: <30 sec) must be measured in the first 30 days post-launch per domain. For high-stakes domains (Finance, HR exceptions), usability testing with at least 5 reviewers is required before go-live. | [Andrej Karpathy — Autonomy Sliders](https://x.com/karpathy) · [DeepLearning.AI — Agentic Design Patterns](https://www.deeplearning.ai/the-batch/how-agents-can-improve-llm-performance/) · [→ P12](./section-10#p12) |
| 4️⃣1️⃣ | **The orchestration layer is the primary engineering surface**<br>*Context construction, topology design, tool argument contracts, and boundary justification determine system performance — not model selection beyond the capability gate.*<br>Engineering effort is commonly misallocated toward model selection and prompt prose rather than orchestration quality. Once a model passes the capability test gate (Principle 38), additional model-selection effort delivers diminishing returns. Agent Design Template quality — context scope, tool argument standards, boundary justification, termination logic — is the primary technical review artifact. Sprint allocation must reflect this: orchestration layer design receives the majority of engineering time. | [Andrej Karpathy — Software 3.0 / LLM OS](https://www.youtube.com/watch?v=LCEmiRjPEtQ) · [HuggingFace — Agentic Shift](https://huggingface.co/blog/dcarpintero/design-patterns-for-building-ai-systems) · [→ P13](./section-10#p13) |
| 4️⃣2️⃣ | **Eval harness is a Sprint 0 deliverable — built before or alongside the agent, never retrofitted**<br>*Gate Sprint 1 build start on eval harness completion — not on design approval.*<br>An eval harness built after an agent is deployed has no gate for the first deployment, no regression signal for early iterations, and is designed to match the built agent rather than test it objectively. Minimum Sprint 0 harness: 50-example golden dataset with expected outputs, LLM-as-Judge rubric, and routing accuracy test for every defined domain intent. Sprint 1 build start is blocked until the eval harness is signed off by the domain lead. | [Andrej Karpathy — Software 3.0](https://www.youtube.com/watch?v=LCEmiRjPEtQ) · [→ P14](./section-10#p14) |
| 4️⃣3️⃣ | **Failure mode profile characterised per agent before deployment — not discovered in production**<br>*Beyond overall pass rate: document which input patterns consistently trigger failures for each domain.*<br>The 90% accuracy gate catches average performance but not the shape of the failure surface. LLMs have jagged capability profiles — the failure rate is not uniform across input types. A failure mode characterisation section in the Agent Design Template must document the input patterns most likely to produce failures per domain (ambiguous intent, unusual formats, edge-case entities, maximum argument complexity). This analysis must be completed on the first 50-sample eval run before build begins. | [Ilya Sutskever — NeurIPS 2024](https://www.youtube.com/watch?v=1yvBqasHLZs) · [MAST](https://huggingface.co/blog/ibm-research/itbenchandmast) · [→ P16](./section-10#p16) |
| 4️⃣4️⃣ | **Safety infrastructure complete before any autonomous capability goes live**<br>*Kill switch tested, HITL flows user-tested, content safety active, eval harness at threshold — all four confirmed, not planned, at go-live.*<br>Every new autonomous capability must have its safety controls designed, tested, and confirmed before it is enabled. "Planned" or "in progress" safety items at go-live time mean capability has outrun safety. Mandatory pre-go-live safety checklist: (1) kill switch tested and confirmed; (2) HITL flows user-tested with real reviewers; (3) content safety layers active for the domain; (4) eval harness passing at gate threshold. Safety infrastructure completion is a separate milestone from capability completion in the delivery roadmap — neither can ship without the other. | [Ilya Sutskever — SSI Founding Principles](https://ssi.inc/) · [→ P17](./section-10#p17) |
| 4️⃣5️⃣ | **Maximum reasoning depth bounded as an explicit architectural parameter per agent role**<br>*Unpredictability scales with reasoning depth — agents with open-ended reasoning scope produce outputs whose validity is progressively harder to verify.*<br>An agent with open-ended reasoning scope can generate chains of arbitrary length, producing outputs proportionally harder to validate. Maximum reasoning depth must be defined in the Agent Design Template per role: Gateway (reasoning disabled — classification only); Domain agents (max N steps, defined per domain); Specialist agents (bounded to the specific subtask); Foundry planning nodes (explicit `max_steps` in Workflow YAML). Reasoning scope violation is a monitored metric in App Insights. | [Ilya Sutskever — NeurIPS 2024](https://www.technology.org/2024/12/16/ilya-sutskever-the-more-ai-reasons-the-more-unpredictable-it-becomes/) · [Dwarkesh Patel Interview 2025](https://www.dwarkesh.com/p/ilya-sutskever-2) · [→ P18](./section-10#p18) |
| 4️⃣6️⃣ | **Adversarial alignment test set required before scale-out**<br>*Standard golden-set evals measure performance on the designed-for distribution — adversarial inputs probe alignment at the boundary of intended behaviour.*<br>Apparent alignment on curated test sets can coexist with misalignment on out-of-distribution inputs. An adversarial test set — probing authority boundary violations, scope creep, instruction override attempts, and cross-agent impersonation — is required to characterise alignment. Adversarial alignment tests must be included in the CI/CD eval harness as a mandatory section, not only in the pre-launch red-team exercise. At least one domain pilot must complete a structured alignment red-team before Month 5 scale-out. | [Ilya Sutskever — Weak-to-Strong Generalization](https://openai.com/index/weak-to-strong-generalization/) · [→ P19](./section-10#p19) |
| 4️⃣7️⃣ | **Benchmark-to-deployment generalisation gap explicitly measured and tracked**<br>*If CI eval harness pass rate exceeds production outcome rate by >10pp for two consecutive weeks, trigger an eval harness review.*<br>A growing gap between CI eval performance and production performance on the same task categories is a concrete signal that the eval harness is no longer representative of the production distribution — but without explicitly tracking this delta, the signal is invisible until degradation is severe. A "generalisation gap" metric must be on the production monitoring dashboard per domain, reviewed in the monthly eval re-run alongside absolute performance metrics. | [Ilya Sutskever — NeurIPS 2024](https://dlyog.com/papers/one_internet_v1) · [Dwarkesh Patel Interview 2025](https://www.dwarkesh.com/p/ilya-sutskever-2) · [→ P20](./section-10#p20) |
| 4️⃣8️⃣ | **Foundation model updates trigger full eval re-run — model version is a tracked dependency**<br>*Managed provider model updates silently change agent behaviour; revalidation within 48 hours of a model update notification is required.*<br>Copilot Studio and Azure AI Foundry update underlying foundation models without breaking API changes — but agent behaviour can change. Model version must be a tracked field in the Agent Design Template and Agent 365 inventory. A model version change has the same revalidation requirement as a prompt change. Provider model update announcements must be integrated with the CI/CD pipeline so a model update notification triggers a scheduled eval run, not a manual review ticket. | [Ilya Sutskever — SSI / Dwarkesh Patel Interview](https://www.dwarkesh.com/p/ilya-sutskever-2) · [→ P21](./section-10#p21) |
| 4️⃣9️⃣ | **Fine-tuning approved only after demonstrable in-context performance gap**<br>*Measure in-context performance first; fine-tune only when the in-context pass rate demonstrably cannot reach the required reliability threshold.*<br>Fine-tuning without this gate creates maintenance burden without proportional reliability gains. The Agent Design Template must include a fine-tuning decision section: document the in-context performance pass rate against the capability test threshold (Principle 11), the performance gap, the available domain data, and the revalidation plan before fine-tuning is approved. Fine-tuning decisions are reviewed at the Agent Design Template gate — not made independently by the build team. | [Sebastian Raschka — Using and Finetuning Pretrained Transformers](https://magazine.sebastianraschka.com/p/using-and-finetuning-pretrained-transformers) · [Practical Tips for Finetuning LLMs](https://magazine.sebastianraschka.com/p/practical-tips-for-finetuning-llms) · [→ P22](./section-10#p22) |
| 5️⃣0️⃣ | **RAG retrieval quality evaluated independently from generation quality**<br>*Retrieval metrics (recall@K, precision, grounding rate, hallucination rate) tracked separately from generation quality with independent thresholds.*<br>A system that retrieves poor documents but generates fluent text passes generation quality evals while failing on factual grounding — producing confident, coherent, but incorrect responses. Retrieval quality is the primary reliability surface for all RAG agents. The CI/CD eval harness and production monitoring must track retrieval metrics independently. Minimum threshold before go-live: retrieval recall@3 >80% (adjust based on pilot data). Improving the generator without measuring retrieval is misallocated engineering effort. | [Sebastian Raschka — Beyond Standard LLMs](https://magazine.sebastianraschka.com/p/beyond-standard-llms) · [→ P24](./section-10#p24) |
| 5️⃣1️⃣ | **Model validated for multi-turn coherence before approval for agentic roles**<br>*Single-turn benchmark scores do not predict multi-turn agentic performance; test at maximum expected session length, not average.*<br>Linear attention architectures fail on multi-turn reasoning tasks foundational to agentic applications. A model can score well on single-turn benchmarks while failing on multi-turn coherence. Agent Design Template model selection must include: (1) multi-turn coherence test at maximum expected session length; (2) structured output/tool call validation with the specific tool schemas for that domain; (3) reasoning depth validation at the maximum chain length expected for the role. Flag linear attention architectures as requiring additional validation before agentic approval. | [Sebastian Raschka — State of LLMs 2025](https://magazine.sebastianraschka.com/p/state-of-llms-2025) · [Beyond Standard LLMs](https://magazine.sebastianraschka.com/p/beyond-standard-llms) · [→ P25](./section-10#p25) |
| 5️⃣2️⃣ | **Eval methodology reviewed bi-annually — methodology can become unrepresentative without agent performance changing**<br>*Review whether the golden dataset, judge model, and evaluation approaches remain representative every 6 months.*<br>Continuous production evaluation catches performance drift against a fixed methodology. It does not catch methodology drift — the harness becoming unrepresentative as the input distribution shifts or better evaluation methods emerge. Bi-annual eval methodology review is a governance deliverable: (1) Is the golden dataset still representative? (2) Is the judge model still best-in-class? (3) Should new evaluation methods be incorporated? (4) Have production failures exposed eval blind spots? The generalisation gap metric (Principle 47) is an early warning trigger for unscheduled reviews. | [Sebastian Raschka — LLM Research Papers 2024](https://magazine.sebastianraschka.com/p/llm-research-papers-the-2024-list) · [→ P26](./section-10#p26) |
| 5️⃣3️⃣ | **Open-weight vs. hosted API decided by evidence — data residency, volume, and fine-tuning requirements**<br>*Not team preference, not perceived capability; three structured conditions determine when open-weight deployment is appropriate.*<br>The architecture defaults to hosted API models but does not preclude open-weight deployment via Azure ML. Open-weight is appropriate when: (1) data residency requirements preclude sending data to a hosted API; (2) call volume is high enough that Azure ML hosting costs are lower than API costs at scale; or (3) a specific fine-tuning requirement requires model weight access. The Agent Design Template must include a structured decision gate for these three conditions. Model source, licence compliance, and security review are required before any open-weight model is approved. | [Sebastian Raschka — Finetuning Large Language Models](https://magazine.sebastianraschka.com/p/finetuning-large-language-models) · [→ P27](./section-10#p27) |
| 5️⃣4️⃣ | **Topology selected from a defined taxonomy — ad-hoc topology is an anti-pattern**<br>*Choose from: hierarchical, sequential, parallel, handoff, group-chat, or Magentic-supervisor; document the choice and its justification.*<br>Agents assembled without a deliberately chosen topology produce unpredictable reliability at scale. Ad-hoc topology — where structure emerges from implementation decisions rather than design decisions — is not auditable, not testable, and not adjustable without re-architecture. The Agent Design Template must record the topology pattern selected for the agent cluster, the justification, and the known reliability characteristics of that pattern. A topology change is a design-level change requiring Architecture review — not an implementation decision. | [Google ADK — Multi-Agent Patterns](https://developers.googleblog.com/developers-guide-to-multi-agent-patterns-in-adk/) · [IBM Guide to Agentic AI Systems](https://www.ibm.com/think/architectures/patterns/agentic-ai) · [→ P54](./section-10#p54) |
| 5️⃣5️⃣ | **A2A and MCP are the interoperability layer — open standards, no proprietary lock-in at the orchestration level**<br>*Agent-to-agent calls use A2A protocol; tool integration uses MCP; both built on open standards to enable platform-agnostic connectivity.*<br>Proprietary point-to-point integrations accumulate coupling that cannot be replaced at scale. The interoperability layer must be built on open standards: A2A for cross-platform agent-to-agent calls (CS → Foundry, Foundry → Agentforce, any external agent); MCP for tool integration across agent runtimes. Vendor lock-in at the orchestration level must be avoided by design. Platform-native connectors are acceptable within a platform boundary; cross-platform calls must use A2A. | [Microsoft Azure Blog — MCP and A2A](https://azure.microsoft.com/en-us/blog/agent-factory-connecting-agents-apps-and-data-with-new-open-standards-like-mcp-and-a2a/) · [IBM Developer — MCP Architecture Patterns](https://developer.ibm.com/articles/mcp-architecture-patterns-ai-systems/) · [→ P55](./section-10#p55) |
| 5️⃣6️⃣ | **Memory, tools, and planning are distinct architectural layers with independent design decisions**<br>*Design decisions for memory (what state to preserve), tools (how to interact with systems), and planning (how to sequence steps) must be made independently — conflating them creates agents that cannot be maintained or upgraded along each dimension.*<br>Memory design drives storage technology selection (Dataverse, Redis, CosmosDB). Tool design drives connector and MCP schema decisions. Planning design drives orchestration pattern and topology selection. Each layer must be defined independently in the Agent Design Template before build begins — not discovered during implementation. Conflating them in a single prompt produces agents with undefined failure boundaries. | [HuggingFace — Building Blocks of Agentic Systems](https://huggingface.co/blog/Kseniase/buildingblocks) · [IBM — What Is Agentic Architecture](https://www.ibm.com/think/topics/agentic-architecture) · [→ P56](./section-10#p56) |
| 5️⃣7️⃣ | **Selective reflection applied to quality-sensitive outputs only — universal reflection wastes latency without proportional benefit**<br>*Reflection must be reserved for outputs where quality failure has high consequence and where the failure mode is detectable by a second LLM pass.*<br>Universal application of reflection patterns (critique → revise → re-evaluate) adds latency and LLM cost to every output without proportional quality benefit. The Agent Design Template must explicitly classify each output type: reflection required, reflection optional, or reflection disabled. Reflection classification is a design decision reviewed at the Agent Design Template gate — not left to default-on. | [DeepLearning.AI — Agentic Design Patterns Part 2: Reflection](https://www.deeplearning.ai/the-batch/agentic-design-patterns-part-2-reflection/) · [→ P57](./section-10#p57) |
| 5️⃣8️⃣ | **Governance infrastructure established before scaling — cannot be retrofitted**<br>*Agent registry, access controls, ALM pipelines, kill switch procedures, and domain isolation must all be operational before the second domain goes live.*<br>Governance added after an agent estate is scaled requires re-architecting each deployed agent. Domain isolation, bounded context, independent ALM pipelines, and agent registry must be designed and validated during the pilot wave (Months 3–4) before scale-out begins. Governance and safety are first-class architectural inputs at inception. The CoE must certify that all governance infrastructure is operational before Month 5 scale-out is approved. | [Microsoft CAF — Build Secure Process](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ai-agents/build-secure-process) · [Microsoft ISE Dev Blog — Multi-Agent at Scale](https://devblogs.microsoft.com/ise/multi-agent-systems-at-scale/) · [→ P58](./section-10#p58) |
| 5️⃣9️⃣ | **Factory build process established before scaling to 100+ agents**<br>*Standard templates, cloneable ALM pipelines, and domain-team onboarding must exist before agent #15 — not built in parallel with scale-out.*<br>Ad-hoc per-agent builds cannot scale to 100+ agents. | 6️⃣0️⃣ | **Inference cost defined as an acceptance criterion alongside quality — not optimised post-deployment**<br>*Latency p50/p95, throughput at peak QPS, and estimated monthly cost at target scale must be declared in the Agent Design Template before build begins.*<br>Treating inference cost as an operational afterthought produces architectures that are correct in design but undeployable at production scale due to token costs, latency limits, or budget overruns. | 6️⃣1️⃣ | **Lightweight input classifier routes to appropriate model tier before LLM invocation**<br>*Route routine extraction tasks to smaller, faster models; reserve frontier reasoning models for inputs that demonstrably require them.*<br>Using a frontier reasoning model for simple extraction, classification, or formatting tasks wastes cost without quality benefit. A lightweight input classifier — rule-based or a small classification model — should determine input complexity class before any large LLM invocation: routine/extraction → smaller model, standard generation → medium model, complex reasoning → frontier model. The classifier must be near-zero cost. The input complexity distribution per domain agent must be measured during the pilot wave to size the investment in tiered routing. | [Noam Shazeer — MoE / conditional compute principle](https://arxiv.org/abs/1701.06538) · [Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns) · [→ P30](./section-10#p30) |
| 6️⃣2️⃣ | **Agent interfaces designed for capability discovery — not hardcoded to current model limitations**<br>*Document current-model limitation workarounds explicitly so they can be removed cleanly when the underlying limitation is resolved by a model update.*<br>Architectures built around today's model limitations (instruction length, reasoning depth, tool-calling reliability) become binding constraints when models improve. Limitation workarounds must be documented in the Agent Design Template so they can be removed cleanly when resolved. Agent interfaces (tool signatures, authority scopes, I/O contracts) must be model-agnostic — not assume the capability level of the current underlying model. Agent capability declarations must be reviewed quarterly as the model landscape evolves. | [Noam Shazeer — a16z Interview](https://a16z.com/universally-accessible-intelligence/) · [Dwarkesh Podcast — Jeff Dean & Noam Shazeer](https://www.dwarkesh.com/p/jeff-dean-and-noam-shazeer) · [→ P31](./section-10#p31) |
