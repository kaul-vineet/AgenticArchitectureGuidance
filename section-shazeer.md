## 🎓 Academic Analysis — Noam Shazeer

*This page maps each of the 28 Key Architectural Principles against Noam Shazeer's published positions on AI systems engineering. Shazeer is co-author of the original Transformer paper, inventor of Multi-Query Attention, Mixture of Experts gating, SwiGLU, and the CharacterFlywheel production evaluation methodology. His work provides the deepest published treatment of inference efficiency, context management at scale, and production-grade evaluation in the field.*

> **Shazeer's core thesis ([deeplearning.ai interview](https://www.deeplearning.ai/the-batch/ai-transformed/) · [Hot Chips 2025](https://hc2025.hotchips.org/assets/program/conference/day1/k1_GoogleDeepMind_Shazeer.pdf)):** Arithmetic is cheap; moving data is expensive. Every architectural decision — attention pattern, expert routing, quantisation, context length — must be co-designed with the hardware reality it runs on. Specialisation at scale is achievable within unified systems through conditional activation, not through architectural fragmentation. Production is the laboratory: fifteen model versions, seven-day A/B tests, and continuous engagement metrics are the engineering method.

The paper explicitly defines two phases: "pre-launch" (V1–V7) and "post-deployment" (V8–V15), with the post-deployment phase using live production traffic as the primary training and evaluation signal. The evaluation is continuous: "integrates data curation, reward modeling, SFT, RL, and both offline and online evaluation to ensure reliable progress at each optimisation step." | CoE Toolkit tracks weekly: intent resolution rate, connector success rate, session abandonment, escalation rate. Monthly: full eval harness re-run to catch silent prompt drift. The CharacterFlywheel pattern is the template: define "engagement breadth" and "engagement depth" equivalents for your domain — what does user success look like in production, and is it being continuously, automatically measured? | [CharacterFlywheel](https://arxiv.org/abs/2603.01973) · [Character.AI Inference Blog](https://research.character.ai/optimizing-inference/) |
| 2️⃣7️⃣ | **Maintain agent inventory — shadow agents are a risk** | <span class="adh-partial">➖ Implied</span> | Shazeer has not published on agent inventory management. However, his architectural consolidation — thousands of personas from three shared model sizes — implicitly eliminates the shadow agent risk at the infrastructure level. If all personas operate from the same Kaiju model family with the same safety architecture, there are no shadow agents with separate deployment lifecycles or separate safety postures. The risk of shadow agents is partly a consequence of agent proliferation; Shazeer's architectural choices prevent proliferation by design. | Agent 365 is the system of record — every CS agent registered before deployment; shadow agents treated as security incidents. The Shazeer design-level insight: centralised model infrastructure with persona differentiation through conditioning reduces the inventory management problem to managing character card configurations, not deployed model instances. For enterprises that cannot fully consolidate, the character card registry (name, persona, scope, safety classification) is the minimum viable inventory. | [Inside Kaiju](https://blog.character.ai/inside-kaiju-building-conversational-models-at-scale/) |
| 2️⃣8️⃣ | **Externalize and version-control prompts** | <span class="adh-partial">🟡 Partial</span> | The Kaiju architecture uses **character cards** as the mechanism for persona differentiation — each of thousands of characters is defined by an externally managed character definition, separate from model weights, updatable without retraining. This is externalised prompts at production scale. CharacterFlywheel tracks instruction-following capability (59.2% → 84.8%) across model versions — this only makes sense if the character card (the instruction/prompt) is externalised and stable enough to serve as a consistent benchmark across model updates. The character card is the enterprise system prompt: it lives outside the model, can be versioned independently, and is testable against an automated eval harness. | CS: Agent instructions stored in solution environment variables — not hardcoded in Topics. Foundry: system prompts in Azure App Configuration. Every prompt change requires PR, reviewer sign-off, and eval harness pass before merge. A prompt change is a code deployment. | [Inside Kaiju](https://blog.character.ai/inside-kaiju-building-conversational-models-at-scale/) · [CharacterFlywheel](https://arxiv.org/abs/2603.01973) |

### ❌ Principles Not in the 28 — Added to Design Considerations

The following 5 principles emerge from Shazeer's published work and have no direct counterpart in the 28 Key Architectural Principles. They have been added to [Design Considerations](./section-10.html) as DC-28 through DC-32.

| ID | Missing Principle | Shazeer's Position | Gap in Your Architecture |
|---|---|---|---|
| DC-28 | **Hardware-algorithm co-design is a first-class engineering discipline** | "Arithmetic is cheap and moving data is expensive on today's hardware." Every architectural choice — context length, attention pattern, quantisation, model tier — must be co-designed with the hardware reality it runs on. MQA, MoE, cross-layer KV sharing, and int8 native training are all consequences of a single axiom: optimise for memory bandwidth, not compute. | Enterprise architectures that finalise agent design in isolation from infrastructure constraints will discover deployment-blockers post-design. KV cache pressure, memory bandwidth limits, and GPU memory constraints must be modelled at the design phase. |
| DC-29 | **Inference cost must be a primary training objective, not an afterthought** | Kaiju is natively trained in int8 precision to eliminate training/serving mismatch. Serving efficiency targets drove the design philosophy — not academic benchmarks. The result: 33x cost reduction at constant quality over two years. Cost is a training constraint, not a post-deployment optimisation. | Custom fine-tuning or model commissioning for enterprise agents should specify inference cost targets (latency, memory footprint, throughput) as acceptance criteria alongside quality benchmarks. Post-training quantisation is an inferior substitute for native-precision training. |
| DC-30 | **Conditional compute — activate only what the input requires** | MoE: only the expert parameters relevant to each input are activated per token. Switch Transformer: K=1 routing activates one expert, not two — the marginal benefit does not justify the routing overhead. Large capacity available; small capacity consumed per inference. Every input should consume the minimum viable capability to produce the required output. | Design agentic routing with a lightweight classifier that determines input complexity before invoking any LLM. The classifier routes to the appropriate model tier (gateway/domain/specialist) at near-zero cost. This is the enterprise-level implementation of MoE conditional compute. |
| DC-31 | **Scale produces emergent capabilities — design interfaces for capability discovery, not just current capability** | "Just make the thing smarter — theory of mind is one of these emergent properties that just is going to come with scale." The right architectural posture is to design for capability emergence: agent interfaces, tool signatures, and authority scopes should be model-agnostic and not assume the capability level of the current underlying model. | Use capability discovery patterns — agents declare what they can do through structured declarations, not hardcoded capability lists. As the underlying model improves, the architecture should automatically benefit without redesign. |
| DC-32 | **Serving infrastructure is a competitive moat — model and serving architecture must co-evolve** | Character.AI's 13.5x cost advantage over leading commercial APIs is not from model quality alone — it is the product of co-designing model architecture (MQA, cross-layer KV sharing, int8) with serving infrastructure (sticky sessions, per-dialogue KV cache, fleet routing). The efficiency is inseparable from the co-design: you cannot implement cross-layer KV sharing if the model was not trained to support it. | When selecting models for production deployment, treat serving compatibility as a primary selection criterion: MQA or GQA (not MHA), quantisation-aware trained at target precision, designed for the context length your use case requires. Require model providers to disclose serving efficiency characteristics as part of model selection criteria. |
