## 🎓 Academic Analysis — Sebastian Raschka

*This page maps each of the 28 Key Architectural Principles against Sebastian Raschka's published positions on LLM engineering, fine-tuning, reasoning, and agentic systems. Raschka's frameworks — LLM engineering from first principles, fine-tuning decision theory, the four evaluation approaches, and the "State of LLMs" annual assessments — provide independent empirical grounding from a practitioner-researcher perspective for the engineering discipline built into this document.*

> **Raschka's core thesis ([Build a Large Language Model From Scratch, Manning 2024](https://sebastianraschka.com/llms-from-scratch/) · [Ahead of AI newsletter](https://magazine.sebastianraschka.com/)):** Understanding LLMs from first principles — architecture, training, fine-tuning, and evaluation — is the foundation for making reliable engineering decisions. The primary risks in production LLM systems are not model capability gaps but evaluation gaps: teams that cannot measure system behaviour cannot improve it, cannot gate deployments reliably, and cannot detect regressions. The engineering discipline around LLMs is still forming; practitioner-grade understanding of internals separates teams that build reliable systems from teams that build fragile ones.

Foundry: system prompts in Azure App Configuration — not in Workflow YAML. Every prompt change requires a PR with reviewer sign-off and must pass the eval harness before merge. A prompt change is a code deployment. | [Finetuning Large Language Models](https://magazine.sebastianraschka.com/p/finetuning-large-language-models) · [Understanding the 4 Main Approaches to LLM Evaluation (From Scratch)](https://magazine.sebastianraschka.com/p/llm-evaluation-4-approaches) |

### ❌ Principles Not in the 28 — Candidates from Raschka's Framework

The following 6 principles emerge from Raschka's published work but are not captured in the 28 Key Architectural Principles. They are proposed as DC-22 through DC-27 for addition to [Design Considerations](./section-10.html).

| ID | Missing Principle | Raschka's Position | Gap in Your Architecture |
|---|---|---|---|
| DC-22 | **Fine-tuning readiness: define when fine-tuning is appropriate per agent role** | Raschka's fine-tuning decision framework is explicit: always test in-context performance first; fine-tune only when in-context performance is demonstrably insufficient. For enterprise agents, this creates a principled decision gate for each agent role: is the agent's current performance (with prompting and RAG) sufficient, or is domain-specific fine-tuning required? Without this gate, teams either over-invest in fine-tuning (unnecessary cost, maintenance burden) or under-invest (insufficient reliability for high-stakes domains). | Add a "fine-tuning decision" section to the Agent Design Template: (1) has in-context performance been measured against the capability test threshold? (2) if not, document the in-context pass rate and its gap from the required threshold; (3) if fine-tuning is proposed, document the domain data available, the target improvement, and the LoRA configuration. Fine-tuning must be justified by a documented performance gap — not by a team preference for model specialisation. |
| DC-23 | **LLM-as-Judge must be calibrated before use as an automated gate** | Raschka's evaluation framework explicitly flags LLM-as-Judge biases (positional, verbosity, self-enhancement) and states that the judge must be calibrated against human annotations before being used as an automated quality gate. DC-6 in Design Considerations already identifies this gap — but Raschka's framework adds specificity: calibration must achieve a measurable human annotation agreement threshold (he suggests >80%) and must be repeated when the judge model is updated. A miscalibrated judge can produce systematically wrong quality signals, creating false confidence before production promotion. | Extend DC-6 (LLM-as-Judge Calibration) with Raschka's specific calibration protocol: (1) human annotation of N=200 samples per domain before the judge is used as an automated gate; (2) agreement threshold of >80% between judge and human annotators; (3) recalibration when the judge model is updated or when production distribution shifts by >10%. Document the calibration results in the Agent Design Template before the judge is used in CI/CD. |
| DC-24 | **RAG retrieval quality requires independent evaluation, separate from generation quality** | Raschka's LLM engineering framework treats RAG as a two-stage system: retrieval quality and generation quality are independent dimensions, and both must be evaluated. A system that retrieves poor documents but generates fluent text will pass generation quality evals while failing on factual grounding. Current evals (Principle 25) measure end-to-end output quality but may not separately measure retrieval quality. For agents that rely on RAG for domain knowledge, retrieval quality is the primary reliability surface — improving the generator without improving retrieval is misallocated effort. | Add RAG-specific evaluation to the Agent Design Template for all agents using retrieval: (1) retrieval recall@K — is the relevant document in the top-K retrieved results? (2) retrieval precision — are the retrieved documents relevant? (3) grounding rate — what proportion of agent assertions are traceable to retrieved documents? These metrics must be tracked independently from generation quality metrics (response relevance, coherence) in the CI/CD eval harness. |
| DC-25 | **Model architecture validation for agentic task types** | Raschka documents that linear attention models fail on multi-turn reasoning tasks — both of which are foundational to agentic use. This is a model architecture concern, not a model size concern. Not all models, regardless of benchmark performance, are suitable for agentic patterns. Before any model is approved for an agentic role (gateway routing, domain reasoning, specialist execution), its architecture must be validated for the specific agentic task type: multi-turn coherence, tool call generation, and reasoning depth. | Add model architecture validation to the Agent Design Template model selection section: for any model approved for an agentic role, document: (1) does the architecture support multi-turn coherence at the expected session length? (2) has structured output / tool call generation been validated with the tool schemas used in this domain? (3) has reasoning depth been tested at the maximum chain length expected for this role? Benchmark scores are not sufficient — architecture-specific tests must be run. |
| DC-26 | **Evaluation methodology must evolve with the agent fleet — set a review cadence** | Raschka's annual "State of LLMs" reports show that evaluation best practices shift materially year-over-year: benchmarks become saturated, new biases in LLM-as-Judge are discovered, and new task categories emerge that prior eval frameworks did not cover. An eval harness designed in Q1 of Year 1 may not be adequate by Q1 of Year 2 — not because the agents have changed but because the evaluation methodology has been superseded. Continuous production evaluation (Principle 26) catches performance drift; it does not catch methodology drift. | Add a bi-annual "eval methodology review" to the governance calendar: every 6 months, review whether: (1) the golden dataset remains representative of the current production input distribution; (2) the LLM-as-Judge model is still the best available for the task type; (3) new evaluation methods from the research community should be incorporated; (4) any eval blind spots have been identified through production failures. The eval methodology review is a governance deliverable alongside the monthly performance review. |
| DC-27 | **Open-weight model strategy: define when open-weight models are appropriate vs. hosted APIs** | Raschka is an active advocate for open-source and open-weight AI — his research, book, and newsletter consistently cover open-weight models as a first-class alternative to hosted APIs. His practitioner framework implies a structured decision: open-weight models offer data residency, fine-tuning flexibility, and cost control at the expense of maintenance burden and infrastructure requirement. For enterprise deployments with data sovereignty requirements or high-volume specialist tasks, open-weight models (deployed via Azure ML or AKS) may be more appropriate than hosted API models. This decision is absent from the current architecture. | Add an "open-weight vs. hosted API" decision to the Agent Design Template model selection section: (1) does this domain have data residency requirements that preclude sending data to a hosted API? (2) is the expected volume high enough that hosting costs favour an open-weight deployment? (3) is there a specific fine-tuning requirement that can only be met with an open-weight model? If any answer is yes, evaluate an open-weight deployment via Azure ML. Document the decision and its rationale in the Agent Design Template. |
