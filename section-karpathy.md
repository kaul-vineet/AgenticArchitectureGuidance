## 🎓 Academic Analysis — Andrej Karpathy

*This page maps each of the 62 Key Architectural Principles against Andrej Karpathy's published positions on AI systems engineering. Karpathy's frameworks — Software 3.0, LLM OS, context engineering, march of nines, and jagged intelligence — provide independent empirical grounding for the architectural choices made in this document.*

> **Karpathy's core thesis ([Software 3.0](https://www.youtube.com/watch?v=LCEmiRjPEtQ)):** LLMs are a new computing primitive. The engineering discipline around them is still forming. The primary risks are not model capability — they are context quality, compound reliability collapse, and premature autonomy.

| 1️⃣ | **Minimise LLM calls** | <span class="adh-full">✅ Full</span> | Each LLM call introduces a probabilistic step; compound calls produce compound error. | 3️⃣ | **Clarification before action** | <span class="adh-full">✅ Full</span> | Karpathy's **"context engineering"** thesis: the quality of the context window is the primary determinant of output quality. Ambiguous input is an under-specified context; acting on it guarantees the model will hallucinate structure to fill the gap. He argues "prompt engineering" is too narrow — the entire context, including all preceding turns, must be managed as a first-class engineering artifact. Ambiguity is not a prompt problem; it is a context problem. | Mandatory disambiguation CS Topic at Gateway and Domain layers before any agentic branch. Ambiguity is resolved, not routed. | 5️⃣ | **Externalize termination logic** | <span class="adh-full">✅ Full</span> | Karpathy's **"LLM OS"** framing: the LLM is the CPU, but CPUs do not govern their own execution lifecycle — the operating system does. Termination, scheduling, and resource limits are OS-layer concerns. Delegating these to the LLM is equivalent to asking the CPU to schedule itself: it will run until it crashes, loops indefinitely, or exhausts resources. The OS governs the CPU; the orchestration layer governs the LLM. | `max_steps`, loop detection, and session time limits are enforced in CS platform settings and Foundry Workflow YAML — never in system prompts. Any agent without externally enforced termination is not approved for production. | [LLM OS framing — Lex Fridman Podcast](https://www.youtube.com/watch?v=cdiD-9MMpb0) |
| 6️⃣ | **Tool argument quality is the #1 failure cause** | <span class="adh-full">✅ Full</span> | Core **"context engineering"** thesis: what enters the context window is the primary engineering surface. Karpathy explicitly identifies argument quality as the most overlooked failure mode in agentic systems — it compounds at every tool call, not just at the boundary. | Every PA connector and Azure Function must include: (1) example call, (2) RFC3339 datetime formats, (3) typed enums, (4) structured error response with field name and expected format. | 7️⃣ | **Consolidate tools** | <span class="adh-partial">🟡 Partial</span> | Karpathy advocates for minimising generative steps; his "march of nines" applies to tool calls as much as to agent hops. He does not specifically publish tool consolidation patterns, but the reliability math is direct: two tools that always execute in sequence share a single error budget and should share a single tool boundary. Two probabilistic calls where one would suffice is an avoidable accuracy penalty. | PA flows that always execute in sequence (check balance + submit leave) are merged into one flow. Azure Functions consolidate multi-step operations with no intermediate branching. Rule: if two calls always execute sequentially with no branch between them, they must be merged. Exception: independent retry or verification is required. | [Intro to LLMs](https://www.youtube.com/watch?v=zjkBMFhNj_g) |
| 8️⃣ | **Multi-agent is a reliability tax** | <span class="adh-full">✅ Full</span> | **"March of nines"** is the direct quantification of this principle. Each agent hop is a probabilistic step; compound accuracy degrades geometrically. Karpathy argues multi-agent systems require explicit reliability budgeting — every boundary must be justified against its cost in end-to-end accuracy. "Don't add an agent hop you don't need." The burden of proof is on the team adding the boundary, not on the team questioning it. | New agent boundaries require one of: different data sovereignty, separate compliance scope, or connectors that cannot be shared. Organisational reasons are explicitly invalid. Each Workflow sub-agent hop adds LLM inference cost and accuracy penalty; boundaries added only where parallel execution or specialist isolation is demonstrably required. | [Intro to LLMs](https://www.youtube.com/watch?v=zjkBMFhNj_g) · [Software 3.0](https://www.youtube.com/watch?v=LCEmiRjPEtQ) |
| 9️⃣ | **Context hygiene at scale** | <span class="adh-full">✅ Full</span> | **"Context engineering"** is Karpathy's primary contribution to agentic systems thinking. The context window is the entire product: what is in it, what is excluded, how it is structured, and when it is compacted determines output quality far more than model selection or prompt phrasing. "We spend too much time on the model and too little on the context." Bloated context is not a minor inefficiency — it is a primary accuracy degradation mechanism. | Gateway passes only intent, userId, and correlationId — no full chat history forwarded. Context passed at each Workflow boundary is fully typed, minimal, and scope-limited. | [Context Engineering (X / various 2024–25)](https://x.com/karpathy) |
| 🔟 | **Sensitive state in side channel** | <span class="adh-partial">🟡 Partial</span> | Karpathy addresses prompt injection as the primary agentic security surface (see Missing Principle F below). Sensitive data in the context window is an injection amplifier: credentials and PII in context are accessible to any injected instruction — the attacker who controls one tool response can exfiltrate everything in context. While Karpathy does not frame this as "side channel architecture," the security implication is identical: secrets in context are secrets available to attackers. | Credentials live in Power Platform connection references — never in Topics or system prompts. PII (employee IDs, balances, salary figures) is fetched at tool execution time and never stored in conversation context between turns. Managed Identity for all Foundry service calls; secrets in Azure Key Vault. | [Prompt injection concerns (X / various)](https://x.com/karpathy) |

Karpathy uses the compound accuracy model explicitly: a 5-tool chain at 95% per step yields 77% end-to-end; at 90% per step it yields 59%. Individual tool accuracy targets must be back-calculated from the required end-to-end accuracy — they cannot be set as a flat 90% regardless of chain length. The math must be done per deployment. | CoE Toolkit and App Insights track per-connector success rate. "The question is not whether to include humans — it is how to design the review interface so that human oversight is genuinely effective and not merely nominal." Bad UX makes HITL a checkbox, not a control. | Three-tier action classification (Advisory / Supervised / Autonomous). | 1️⃣3️⃣ | **Use lowest complexity that reliably works** | <span class="adh-full">✅ Full</span> | Karpathy's explicit position: "The best code is no code; the best LLM call is no LLM call." He frames over-engineering with LLMs as a form of technological solutionism — reaching for the most powerful tool when a simpler one is correct and sufficient. His Software 3.0 thesis argues for matching the computing primitive (deterministic code vs. LLM) to the task type. | 1️⃣4️⃣ | **Justify every agent boundary explicitly** | <span class="adh-partial">🟡 Partial</span> | Karpathy does not publish specific agent boundary criteria, but his "march of nines" and multi-agent reliability tax positions together imply the need for explicit justification: every boundary is a compounding error source. His "jagged intelligence" observation adds a second concern: each new boundary exposes a new capability surface, each of which must be audited independently. | Three valid reasons for a new boundary: (a) security/compliance isolation, (b) prompt complexity >8,000 chars, (c) tool overload >~15 tools. | 1️⃣7️⃣ | **Validate output before passing downstream** | <span class="adh-full">✅ Full</span> | Karpathy argues LLM output must always be treated as probabilistic — even when it appears well-formed. Validation is the interface contract between the probabilistic world (LLM output) and the deterministic world (downstream processing). Without it, a single hallucinated field can cascade without error signal through the entire agent chain, corrupting downstream state invisibly. Validation is not an optional quality gate; it is a reliability boundary. | JSON schema validation at each Foundry Workflow node; typed YAML validation for CS connected agents. | 1️⃣8️⃣ | **Surface errors explicitly — never swallow failures** | <span class="adh-full">✅ Full</span> | Karpathy explicitly: "Silent failures are the worst failure mode in LLM systems." Silent failures are invisible in production, compound across agent hops, and — because the LLM may cover them with plausible text — are often never discovered until downstream damage is already done. | Every PA connector returns `{errorCode, field, expectedFormat, retryable}`. | 1️⃣9️⃣ | **Idempotency for all writes** | <span class="adh-partial">➖ Implied</span> | Karpathy does not specifically address idempotency in published work, but his reliability engineering framing implies it: retry is a required resilience mechanism in any probabilistic system, and retry without idempotency produces duplicate side effects. His "march of nines" analysis requires retry to achieve the advertised end-to-end accuracy — making idempotency a prerequisite for the reliability model itself to hold. Without it, retried accuracy gains are converted into data integrity losses. | PA flows and Azure Functions include idempotency key `{correlationId}-{actionType}-{date}`; flow checks Dataverse for existing key before executing. | 2️⃣2️⃣ | **Content safety at every layer** | <span class="adh-partial">🟡 Partial</span> | Karpathy focuses on **prompt injection** as the primary attack vector (input safety at the tool response layer), with less specific attention to output content safety classification. | 2️⃣3️⃣ | **Scope agent authority to operations** | <span class="adh-full">✅ Full</span> | Karpathy's **"minimal footprint"** principle: agents must request only the permissions required for the specific task. ### ✅ Principles Incorporated into Key Architectural Principles (P38–P43)

The following 6 principles emerged from Karpathy's published work. They were previously captured as DC-10 through DC-15 and are now incorporated into the 62 Key Architectural Principles as **P38** (task-specific capability test), **P39** (eval distribution statistics), **P40** (HITL as UX), **P41** (orchestration layer as primary engineering surface), **P42** (eval harness Sprint 0), and **P43** (failure mode characterisation before deployment).

| ID | Missing Principle | Karpathy's Position | Gap in Your Architecture |
|---|---|---|---|
| DC-10 | **Capability audit per task — not just model tier** | **Jagged intelligence**: LLMs have uneven capability profiles. A model that scores well on general benchmarks may fail on the specific subtask you need. Right-sizing the model tier (Gateway vs. specialist) is necessary but insufficient without a task-specific capability test. The model must be tested on examples of the actual task — not on benchmarks. | Agent Design Template model selection section must require a pass/fail capability test for the specific subtask, not just a general model tier declaration. "Seems capable" is not a validated choice. |
| DC-11 | **Test prompt distributions, not single runs** | Same prompt produces different outputs across runs due to temperature, context variation, and model version drift. A single successful test run is not a meaningful reliability signal. You must characterise the full output distribution across N samples (≥50) to know where the failure boundary is and what the variance looks like. | The 50-sample eval harness (Principle 25) captures this — but the framing must be explicit in the Agent Design Template: one successful run is not evidence of reliability. Eval results must report distribution statistics (pass rate, variance, failure mode distribution), not just median outcome. |
| DC-12 | **HITL must be engineered as UX, not just inserted as a step** | "Autonomy sliders": human oversight is only real if the review interface makes meaningful review possible. A bad review UX — too much information, insufficient time, no visible undo path — makes oversight nominal: humans approve without actually reviewing. The UX is the control, not the presence of a HITL step. A HITL step with bad UX provides less safety than no HITL step with a narrow permission scope. | PA Adaptive Card approval flows must be explicitly designed for reviewability: action, reasoning, source data, and undo path visible in under 30 seconds. Add to Agent Design Template DoD: measure actual review time and approval-without-review rate per domain. |
| DC-13 | **The orchestration layer is the primary engineering surface** | "Context engineering" and "LLM OS" together: the model is fixed; the engineering surface is the orchestration layer — context construction, topology design, tool argument contracts, and compaction strategy. Effort allocated to model selection over orchestration quality is misallocated. The model is a commodity; the orchestration layer is the competitive differentiator. | Agent Design Template, context hygiene rules, tool argument standards, and boundary justification criteria are the primary engineering deliverables — not model selection or system prompt prose quality. Invest engineering time accordingly: orchestration layer first, model selection last. |
| DC-14 | **Evals must be built before or alongside the agent — not after** | "If you don't have evals, you are flying blind." Karpathy argues the eval harness is a Sprint 0 deliverable. Building agents without an eval framework means you cannot measure progress, cannot set a deployment gate, and cannot detect regressions after model updates or prompt changes. Retrofitting evals to a deployed agent is harder and produces lower-quality coverage than building them alongside. | Eval harness for each domain must be written before the first agent build — not retrofitted after deployment. This is a prerequisite for Principle 25 (automated evals in CI/CD) to work correctly. Add to Agent Design Template: eval harness sign-off is required before build begins, not before go-live. |
| DC-15 | **Prompt injection is the primary agentic security surface** | LLMs have no reliable separation between instruction and content. Every external data source that enters the context window — SharePoint documents, HRIS records, CRM fields, Jira tickets, email attachments — is a potential injection vector. A malicious instruction embedded in a retrieved document will be processed as an agent instruction. Content safety at the output layer does not protect against injection at the tool response layer. | DC-3 (Indirect Prompt Injection) must be treated as a primary security concern, not a secondary design consideration. Response sanitisation between every tool response and LLM context re-entry is a required security layer — not optional. Azure Content Safety prompt shield must be evaluated for all tool responses from user-writable systems before any domain goes to production. |
